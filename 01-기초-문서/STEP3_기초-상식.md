# 1부. 왜 바꿔야 하나 — ChatGPT의 3가지 한계

## 1.1 ChatGPT를 쓰며 겪으셨을 3가지 장면

| 증상 | 현실의 장면 |
|------|------------|
| **맥락 소실** | 어제 쓴 제안서 톤을 오늘 또 처음부터 설명하고 있다 |
| **반복 불가** | 지난달 그 프롬프트를 어디에 뒀는지 모르겠다 |
| **위임 불가** | 결과를 복붙해서 문서에 옮기고 있다 |

이 3가지가 어제의 내 장면이라면, 클로드 코드로 옮길 때입니다.

## 1.2 버려야 할 4가지 습관

ChatGPT 유저일수록 오히려 장애물이 되는 습관입니다.

1. **매번 처음부터 설명하는 습관** → 클로드 코드는 파일에 한 번만 써 두면 됩니다
2. **프롬프트를 채팅창에 날리고 잊는 습관** → 프롬프트는 **자산**입니다. 파일로 저장하세요
3. **결과를 사람이 복붙해서 옮기는 습관** → AI가 직접 파일을 읽고 씁니다
4. **"AI는 틀린다"를 약점으로만 보는 습관** → 틀린다는 전제 아래 **자동 검증 장치**를 답니다

# 2부. 3원리 — 클로드 코드의 뼈대

이 3개만 이해하시면, 나머지는 전부 "어느 원리의 구체물인가"로 분류됩니다.

## 2.1 원리 ① 맥락 (Context)

**정의**: AI가 답하기 전에 읽는 **토대 정보**. 프롬프트(지시)와 구분됩니다.

- 프롬프트 = "이거 해 줘" (지시)
- 컨텍스트 = "참고로 나는 이런 사람이고, 우리 회사 톤은 이렇고, 이 파일을 보고…" (토대)

AI 답변 품질의 대부분은 맥락이 결정합니다. 프롬프트 기술이 아닙니다.

### 맥락 3층 구조

| 층 | 역할 | 예시 파일 |
|----|------|----------|
| **① 영속 맥락** | 매번 자동 로딩 | `CLAUDE.md` (브랜드 톤·상시 규칙·금기) |
| **② 참조 맥락** | 필요할 때 호출 | `brand-guide.md`, `template.md` (긴 문서) |
| **③ 세션 맥락** | 대화 중 잠깐 | 지금 하는 작업의 요구사항 |

> 💡 **가장 큰 오해**: "CLAUDE.md 하나에 다 넣으면 되지" → 커지면 세션마다 로딩 비용이 증가합니다. **3층으로 나눠야 경제적**입니다.

## 2.2 원리 ② 도구 (Tool)

**정의**: AI에게 손발을 주는 방법. "AI가 무엇을 할 수 있는가"를 결정합니다.

### 도구 3층

| 층 | 무엇 | 예시 |
|----|------|------|
| ① **내장 도구** | 클로드 코드에 기본 탑재 | 파일 읽기·쓰기, 터미널 명령, 웹 검색 |
| ② **MCP** | 외부 앱 연결 | Notion, Google Calendar, Slack, GitHub |
| ③ **내가 만든 도구** | 본인 업무 박제 | 주간 회고 스킬, 제안서 초안 커맨드 |

> 🔑 **도구를 만든다 = 파일 하나를 쓴다**. 이것이 이 문서 전체의 해방 메시지입니다.

## 2.3 원리 ③ 검증 (Verification)

**정의**: AI가 틀릴 수 있다는 전제 아래, 자동으로 걸러 내는 장치입니다.

### 검증 4계층 피라미드 (업무 민감도별)

| 층 | 무엇 | 누가 | 적용 예시 |
|----|------|------|----------|
| ④ **수동 검토** | 사람이 눈으로 최종 승인 | 사람 | 채용·법무·발행 |
| ③ **승인 게이트** | 특정 파일이 있어야 진행 | 파일 플래그 | HR·재무 |
| ② **리뷰 서브에이전트** | 두 번째 AI가 첫 AI를 검사 | AI | 콘텐츠 발행 |
| ① **훅(자동 차단)** | 조건 맞으면 자동 차단 | 코드 | 일반 |

> 💡 모두 다 적용할 필요는 없습니다. 개인 메모는 ①만, HR은 ①②③, 채용은 ①②③④ 전부.

# 3부. 작업실 — 폴더가 곧 AI의 뇌

## 3.1 핵심 원리

너저분한 Desktop에 AI를 풀어놓으면, 너저분한 답이 돌아옵니다.
AI의 성능은 폴더 구조에 적잖은 영향을 받습니다.

## 3.2 5폴더 기본형

```
claude-workspace/
├── 01-context/        ← CLAUDE.md, 브랜드 가이드 (영속 맥락)
├── 02-projects/       ← 끝나는 일 (예: Q2 제안서)
├── 03-areas/          ← 계속되는 영역 (회의록·업무·블로그)
├── 04-resources/      ← 참고 자료·템플릿
└── 05-archive/        ← 끝난 것
```

### 폴더 판단 기준 (딱 하나)
- **끝나는가?** → `02-projects/`
- **계속되는가?** → `03-areas/`

### 번호 접두사를 붙이는 이유
- 알파벳 순 정렬 시 원하는 순서가 유지됩니다
- 5개 전부 만들 필요는 없습니다. `01`과 `03`만 만들어도 시작할 수 있습니다

# 4부. CLAUDE.md — AI의 뇌를 복사한다

## 4.1 CLAUDE.md란

**AI가 매번 자동으로 읽는 파일**입니다. 여기에 한 번 써 두면 다시 설명할 필요가 없습니다.

- 루트 폴더(`claude-workspace/`) 또는 `01-context/`에 `CLAUDE.md` 생성
- 클로드 실행 시 자동 로딩
- **v1은 대충, v2에서 정제** — 완벽주의 금지

## 4.2 5섹션 표준 템플릿

```markdown
# CLAUDE.md

## 1. 나는 누구
[이름, 업종, 회사 규모, 역할 3줄]

## 2. 톤·보이스
[우리 브랜드 말투, 피할 표현, 선호 표현]

## 3. 자주 하는 업무
[주간 회고, 제안서, 회의록 요약 … 5개 이상]

## 4. 금기
[민감 정보, 확정 안 된 숫자, 경쟁사 비방 등]

## 5. 도구
[연결된 MCP, 자주 쓰는 스킬 — 처음엔 비워 둠]
```

각 섹션 **3~10줄**이 상한선입니다. 넘치면 별도 파일로 빼서 `@`로 참조합니다.

## 4.3 `@` 레퍼런싱 — 참조 맥락 부르기

긴 문서(브랜드 가이드·템플릿·지난주 회의록)는 CLAUDE.md에 넣지 않고, 별도 파일로 두고 필요할 때만 호출합니다.

```
> 이번 주 블로그 글 초안, @04-resources/brand-guide.md 톤으로 써 줘
```

→ 클로드가 `brand-guide.md`를 그때 읽고, 일이 끝나면 맥락에서 빠집니다. 경제적입니다.

# 5부. 터미널 5명령어

이 강의 끝까지 이 5개만 쓰시면 됩니다. 코드가 아니라 **길 찾기 단어** 5개입니다.

| 명령어 | 의미 | 예시 |
|--------|------|------|
| `cd` | change directory (이동) | `cd ~/Documents/claude-workspace` |
| `ls` | list (이 폴더에 뭐가 있나) | `ls` |
| `mkdir` | make directory (폴더 만들기) | `mkdir 02-projects` |
| `code .` | VS Code로 이 폴더 열기 | `code .` |
| `claude` | 클로드 코드 실행 | `claude` |

공백으로 구분해서 한 번에 여러 개 생성할 수 있습니다.
```bash
mkdir 01-context 02-projects 03-areas 04-resources 05-archive
```

# 6부. 권한 모드 — 3개만 기억

클로드 코드에는 6개 권한 모드가 있지만, 실무에서는 3개만 씁니다.

| 모드 | 용도 | 전환 방법 |
|------|------|----------|
| **default** | 기본. 파일 쓰기 전에 물어봄 | 기본값 |
| **plan** | 읽기만, 쓰지 않음. 계획 짜기 | `Shift + Tab` |
| **acceptEdits** | 파일 쓰기 자동 허용 (빠름) | 메뉴에서 전환 |

> ⚠️ `bypassPermissions` 모드는 쓰지 마세요. 이 강의의 절반(검증 원리)을 버리는 것과 같습니다.

### Plan Mode 사용 타이밍
- 새 작업 시작할 때 → `Shift + Tab`으로 Plan 진입
- "이렇게 할 예정"을 먼저 확인 → OK면 일반 모드로 돌아가 실행

# 7부. MCP — AI에게 외부 앱을 연결

## 7.1 MCP란

**AI와 앱 사이의 USB-C 포트**입니다.

- 이전: AI는 ChatGPT 창 안에서만 작동
- 지금: Notion·캘린더·Slack·GitHub·Gmail에 직접 접근

## 7.2 업무별 우선 연결 Top 5

| 순위 | MCP | 용도 |
|------|-----|------|
| 1 | **Notion** | 위키·회의록·업무 DB |
| 2 | **Google Calendar** | 일정 조회·생성 |
| 3 | **Gmail** | 메일 초안·요약 |
| 4 | **Slack** | 메시지 검색·요약 |
| 5 | **Google Drive** | 파일 검색·내용 조회 |

## 7.3 stdio vs Remote — 보안 핵심

| 종류 | 실행 위치 | 보안 | 적용 |
|------|----------|------|------|
| **stdio** | 내 PC 안 | 🟢 높음 | 민감 데이터 (HR·재무·개인정보) |
| **Remote** | 외부 서버 | 🟡 OAuth 의존 | 일반 데이터 |

> 🔑 HR·재무·개인정보 MCP는 **무조건 stdio**로 갑니다.

## 7.4 도입 전 5가지 체크리스트

- [ ] 어떤 데이터에 접근하는가? (Scope 명시)
- [ ] Read-only로 시작할 수 있는가?
- [ ] OAuth 토큰 저장 위치는?
- [ ] 언제든 revoke 가능한가?
- [ ] 회사 IT 정책 위배는 없는가?

# 8부. 훅 — AI 자동화의 안전벨트

## 8.1 훅이란

**특정 순간에 자동으로 실행되는 규칙**입니다. 엘리베이터 센서·자동차 안전벨트 같은 장치입니다.

- 파일 쓰기 **직전**에 민감 단어가 있으면 차단
- 파일 쓰기 **직후**에 자동 포맷 정리

## 8.2 훅 8종 중 외울 건 3개

| 훅 | 언제 | 용도 |
|----|------|------|
| **PreToolUse** | 도구 쓰기 전 | 민감 단어 차단, 파일 경로 검증 |
| **PostToolUse** | 도구 쓰기 후 | 자동 포맷, 로그 기록 |
| **Stop** | AI 답변 끝날 때 | 최종 검토, 알림 발송 |

## 8.3 설계 원칙

- **비개발자는 알림형부터** 시작하세요 (`echo "[$(date)] wrote: $FILE" >> ~/claude-log.txt`)
- 차단은 명확한 규칙만 넣습니다. 과용하면 AI가 아무것도 못 합니다
- `.claude/settings.json`에서 해당 훅 한 줄 제거로 언제든 해제 가능

# 9부. 서브에이전트 + 승인 게이트

## 9.1 서브에이전트란

**두 번째 AI가 첫 번째 AI의 결과를 검사**하는 구조입니다.

- 첫 AI: 제안서 초안 작성
- 두 번째 AI(reviewer): "숫자 근거 없음·경쟁사 비방 있음" 지적
- 수정 → 다시 검사 → OK → 사람에게

## 9.2 파일 플래그로 사람 승인 넣기

특정 파일이 있어야 AI가 전진하도록 만듭니다.

```
.agent-state/
├── .waiting      ← AI가 대기 중
├── .approved     ← 사람이 "OK" 신호
└── .revise       ← 사람이 "수정 필요" 신호
```

승인 방법:
```bash
touch .approved
```

## 9.3 tools 최소화 원칙

서브에이전트를 만들 때 권한을 최소화합니다.
- reviewer는 **Read-only** (검사만)
- 수정 권한은 **별도 에이전트**로 분리

# 10부. 스킬 / 커맨드 / 에이전트

## 10.1 3개 개념 요약

| 개념 | 무엇 | 언제 | 예시 |
|------|------|------|------|
| **스킬** (SKILL.md) | 절차가 있는 반복 업무 | 매주 회고, 제안서 초안 | "주간 회고 생성" |
| **커맨드** (`/cmd`) | 자주 치는 단축 명령 | 긴 프롬프트 절약 | `/review`, `/summary` |
| **에이전트** | 특화 역할의 AI | 리뷰·감사·특정 도메인 | `reviewer.md`, `bias-auditor.md` |

## 10.2 의사결정 트리

```
Q1. 같은 일을 3번 이상 했는가?
    ├─ NO → 그냥 프롬프트로 처리
    └─ YES → Q2
             Q2. 절차가 있는가?
                 ├─ YES → 스킬
                 └─ NO → Q3
                         Q3. 단축 타이핑이 목적인가?
                             ├─ YES → 커맨드
                             └─ NO → 에이전트
```

> 🔑 **애매하면 스킬**. 80%는 스킬로 해결됩니다.

## 10.3 5가지 안티패턴

| # | 안티패턴 | 왜 위험 | 대체 |
|---|---------|---------|------|
| 1 | **커맨드 남발** | 10개 넘으면 못 찾음 | 스킬로 통합 |
| 2 | **에이전트 Write 권한** | 무한 수정 위험 | Read-only |
| 3 | **SKILL.md 없이 스킬 주장** | 재현 불가 | 반드시 파일로 |
| 4 | **tools 전부 허용** | 보안·제어 구멍 | 필요한 것만 |
| 5 | **민감 데이터 하드코딩** | 유출 시 재앙 | 환경변수·MCP 분리 |
