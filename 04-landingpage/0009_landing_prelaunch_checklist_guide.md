## 랜딩페이지 배포 전 체크리스트 및 실행 가이드

이 문서는 지금까지의 대화를 바탕으로, 아래 조건의 랜딩페이지를 배포하기 전까지 무엇을 어떤 순서로 준비해야 하는지 정리한 실행 문서다.

기준 상황은 다음과 같다.

- 목적: 리드 수집 + 이후 결제 유도
- 가격 공개 방식: 랜딩페이지에서 정확 가격 공개
- 전환 구조: 폼 제출 → 자동 안내 메일 발송 → 메일 또는 Thank you 페이지에서 결제 유도
- 핵심 분석 도구: PostHog
- 기본 검색/발견 가능성 대비: SEO, GEO 체크

---

## 1. 전체 구조 먼저 이해하기

이 랜딩페이지는 단순 소개 페이지가 아니다.
핵심은 아래 퍼널을 운영하고 개선하는 것이다.

```text
유입
→ 랜딩페이지 방문
→ 핵심 메시지 확인
→ CTA 클릭
→ 폼 제출
→ 자동 안내 메일 수신
→ 결제 링크 클릭
→ 결제 완료
```

즉, 배포 전 준비의 핵심은 크게 4가지다.

1. 페이지가 기술적으로 문제 없이 작동해야 한다.
2. 누가 들어와서 어디서 이탈하는지 추적 가능해야 한다.
3. 폼 제출 후 바로 결제까지 이어지는 동선이 있어야 한다.
4. 배포 후 개선할 수 있을 정도로 데이터가 남아야 한다.

---

## 2. 배포 전 준비를 단계별로 보면

권장 순서는 아래와 같다.

### 1단계. 전환 구조 확정
먼저 "무슨 흐름으로 고객을 움직이게 할 것인지"를 확정한다.

확정해야 할 것:
- 핵심 CTA 1개
- 폼에서 받을 항목
- 폼 제출 직후 보여줄 문구와 버튼
- 자동 발송 메일 내용
- 결제 링크 위치
- 가격, 제공 내용, 대상 고객, FAQ

이 단계에서 확정해야 하는 이유:
- 추적 이벤트 설계가 이 구조를 기준으로 달라짐
- PostHog 세팅도 어떤 행동을 추적할지 정해야 의미가 생김
- 페이지 카피와 기능이 서로 어긋나면 전환률이 떨어짐

실행 체크:
- [ ] 핵심 CTA 문구 확정
- [ ] 랜딩페이지 섹션 구조 확정
- [ ] 폼 항목 확정
- [ ] Thank you 페이지 문구 확정
- [ ] 자동 메일 초안 확정
- [ ] 결제 링크 확정

---

### 2단계. 페이지 자체 완성
눈에 보이는 랜딩페이지를 먼저 완성한다.

최소 포함 요소:
- 첫 화면(헤드라인, 서브카피, CTA)
- 대상 고객 설명
- 무엇을 제공하는지
- 가격
- 신청 방식
- FAQ
- 제작자/운영자 신뢰 요소
- 개인정보 처리방침 링크

권장 포인트:
- Above the fold에서 핵심 메시지와 가격이 바로 보여야 함
- CTA는 여러 개 두더라도 모두 동일 행동으로 연결해야 함
- 가격 공개를 했기 때문에 "무엇을 받는지"가 더 명확해야 함
- 폼은 길지 않아야 함

실행 체크:
- [ ] 모바일 반응형 확인
- [ ] CTA 버튼이 모두 정상 작동
- [ ] 가격 및 제공 내용이 모호하지 않음
- [ ] FAQ 최소 3개 이상 작성
- [ ] 개인정보 처리방침 링크 연결

---

### 3단계. 리드 수집 구조 연결
폼이 단순히 예쁘게 보이는 것만으로는 부족하다.
폼 데이터가 실제로 저장되고, 후속 동작이 일어나야 한다.

최소 구조:
- 폼 제출
- 제출 데이터 저장
- 자동 안내 메일 발송
- 내부 알림 수신

권장 수집 항목:
- 이름
- 이메일
- 선택 항목 1개 정도(예: 어떤 점이 가장 궁금한가)

초기에는 추천하지 않는 것:
- 전화번호
- 회사 규모
- 예산
- 긴 자유서술

이유:
- 초기 랜딩은 정보 수집보다 전환률 확인이 먼저다.
- 질문이 많아질수록 제출률이 떨어질 가능성이 높다.

실행 체크:
- [ ] 폼 제출 시 데이터가 시트/DB에 저장됨
- [ ] 자동 안내 메일 발송됨
- [ ] 운영자에게 알림이 감
- [ ] 중복 제출 시 대응 방식 확인
- [ ] 스팸/오입력 최소 방지 장치 적용

---

### 4단계. Thank you 페이지와 결제 유도 구조 세팅
지금 구조에서는 폼 제출 후 메일만 기다리게 하면 전환이 빠질 가능성이 크다.

권장 구조:
- 폼 제출 완료
- "안내 메일을 발송했습니다" 문구 노출
- 바로 아래에 "지금 바로 결제하기" 버튼 노출

즉, 메일은 백업 채널이고,
Thank you 페이지는 즉시 결제 채널이어야 한다.

Thank you 페이지에 들어가면 좋은 것:
- 신청 완료 안내
- 상품/서비스명 재표기
- 가격 재표기
- 바로 결제 버튼
- 문의 방법

실행 체크:
- [ ] 폼 제출 후 Thank you 페이지로 이동
- [ ] Thank you 페이지에 결제 버튼 있음
- [ ] 결제 링크 정상 작동
- [ ] 메일에도 동일 결제 링크 포함

---

### 5단계. PostHog 세팅
이 단계가 매우 중요하다.
배포 전에는 적어도 "누가 어디서 이탈하는지"를 볼 수 있어야 한다.

PostHog를 먼저 설치하고, 핵심 이벤트를 설계한다.

초기 최소 이벤트:
- page_view
- cta_click
- form_start
- form_submit
- thank_you_view
- checkout_click

가능하면 추가:
- scroll_50
- scroll_90
- email_click

실행 체크:
- [ ] PostHog 설치 완료
- [ ] 페이지뷰 수집 확인
- [ ] CTA 클릭 이벤트 확인
- [ ] 폼 제출 이벤트 확인
- [ ] Thank you 페이지 진입 이벤트 확인
- [ ] 결제 링크 클릭 이벤트 확인
- [ ] 세션 리플레이 활성화
- [ ] 퍼널 생성 완료

---

### 6단계. UTM / 유입 채널 추적
배포 후에는 "리드가 생겼다"보다 "어디서 온 리드인가"가 중요해진다.

최소한 아래는 저장돼야 한다.
- utm_source
- utm_medium
- utm_campaign

예시:
- x, threads, blog, newsletter, community 등

이걸 안 남기면 나중에 다음 질문에 답하기 어렵다.
- 어떤 채널이 제일 효율적인가?
- 어떤 콘텐츠에서 전환이 잘 되는가?
- 어디에 시간과 돈을 더 써야 하는가?

실행 체크:
- [ ] 랜딩 유입 URL에 UTM 부착 규칙 정함
- [ ] UTM이 PostHog 또는 리드 데이터에 저장됨
- [ ] 메일 내부 결제 링크에도 UTM 부착

---

### 7단계. SEO / GEO 기본 세팅
이 단계는 "유입 최적화"의 출발점이다.
다만 지금 단계에서는 전환 동선보다 우선순위가 높지는 않다.

최소 체크 항목:
- title
- meta description
- OG title
- OG description
- OG image
- favicon
- robots.txt
- sitemap.xml
- canonical

GEO 관점에서 신경 쓸 것:
- 서비스가 누구를 위한 것인지 문장으로 명확해야 함
- 무엇을 제공하는지 구체적으로 적혀 있어야 함
- 가격과 방식이 분명해야 함
- FAQ 구조가 있으면 좋음

즉, 검색엔진뿐 아니라 AI 요약/추천 맥락에서도 페이지 의미가 명확해야 한다.

실행 체크:
- [ ] title 설정
- [ ] meta description 설정
- [ ] OG 이미지 연결
- [ ] robots.txt 설정
- [ ] sitemap.xml 생성
- [ ] canonical 확인
- [ ] FAQ 작성

---

### 8단계. 에러 트래킹 및 QA
랜딩페이지는 작은 오류 하나로도 리드를 날릴 수 있다.
특히 폼, 메일, 버튼 관련 오류는 반드시 잡아야 한다.

권장 항목:
- 에러 트래킹 도구(Sentry 또는 PostHog 연동 기능)
- 실제 제출 테스트
- 모바일 테스트
- 브라우저 테스트

꼭 해볼 테스트:
- 내가 직접 폼 제출
- 다른 이메일로 메일 수신 확인
- 모바일에서 폼 제출
- Thank you 페이지 이동 확인
- 결제 버튼 클릭 확인

실행 체크:
- [ ] 폼 제출 테스트 완료
- [ ] 자동 메일 수신 테스트 완료
- [ ] 모바일 테스트 완료
- [ ] 크롬 기준 확인
- [ ] Safari/Edge 확인 가능하면 진행
- [ ] 에러 로그 수집 가능 상태

## 3. PostHog 상세 설명

이 문서에서 가장 중요하게 이해해야 할 도구가 PostHog다.

### 3-1. PostHog는 무엇인가
PostHog는 웹사이트나 제품 안에서 사용자가 어떻게 행동하는지 추적하고 분석하는 도구다.

쉽게 말하면,
"누가 왔는지"보다
"와서 무엇을 했고 어디서 멈췄는지"를 보게 해주는 도구다.

GA4가 보통 채널 단위, 마케팅 단위로 많이 쓰인다면,
PostHog는 페이지 안 행동 흐름과 전환 병목을 보는 데 훨씬 직관적이다.

특히 랜딩페이지 검증 단계에서는 아래 질문에 답하는 데 유용하다.

- 사람들이 첫 화면까지는 보는가?
- CTA를 누르는가?
- 폼을 시작하고도 중간에 나가는가?
- 제출은 했는데 결제 버튼은 안 누르는가?
- 모바일에서 이상하게 보이거나 멈추는가?

---

### 3-2. 왜 지금 PostHog가 필요한가
지금 너의 목적은 예쁜 랜딩페이지를 만드는 것이 아니라,
"이 페이지가 실제로 전환을 만드는지"를 확인하는 것이다.

그런데 단순 방문자 수만 보면 거의 아무것도 모른다.

예를 들어 방문자가 100명 왔다고 해도 아래 경우를 구분해야 한다.

- 100명이 왔는데 아무도 CTA를 안 눌렀다
- 100명이 왔고 20명은 CTA를 눌렀지만 폼에서 다 이탈했다
- 100명이 왔고 10명은 폼 제출했지만 Thank you 페이지에서 결제 버튼을 안 눌렀다
- 모바일 사용자만 유독 중간에 떨어진다

이 차이를 모르면 개선 방향이 전혀 안 잡힌다.

즉, PostHog는 단순 분석 도구가 아니라
"무엇을 고쳐야 할지 알려주는 도구"에 가깝다.

---

### 3-3. PostHog를 달면 내가 뭘 할 수 있나
아래가 실제로 가능한 일들이다.

#### 1) 퍼널 보기
예시 퍼널:

```text
page_view
→ cta_click
→ form_submit
→ thank_you_view
→ checkout_click
```

이걸 보면 각 단계에서 몇 %가 다음 단계로 넘어가는지 볼 수 있다.

예를 들어:
- 방문 100명
- CTA 클릭 25명
- 폼 제출 8명
- 결제 클릭 2명

그럼 바로 해석이 가능하다.
- 메시지는 어느 정도 먹힘(25명 클릭)
- 하지만 폼이 부담스럽거나 신뢰가 부족함(8명 제출)
- 결제 버튼 매력이나 urgency가 약함(2명 클릭)

즉, 막연히 "전환이 안 된다"가 아니라
"어디서 전환이 안 되는지"를 볼 수 있다.

#### 2) 세션 리플레이 보기
이 기능은 실제 사용자의 화면 행동을 다시 보는 기능이다.

볼 수 있는 것:
- 어디까지 스크롤했는지
- 어디에서 멈췄는지
- 버튼을 눌렀는지
- 폼 입력하다 멈췄는지

이 기능이 중요한 이유:
숫자만으로는 이유를 모르는 경우가 많다.

예시:
- CTA 클릭률이 낮다 → 왜 낮은지 모름
- 세션 리플레이를 보니 첫 화면 메시지가 길고, 모바일에서 버튼이 너무 아래에 있음

이런 식으로 정성적 원인을 확인할 수 있다.

#### 3) 이벤트 기반 분석
클릭, 제출, 이동 같은 행동을 이벤트로 기록할 수 있다.

예시 이벤트:
- cta_click
- form_start
- form_submit
- checkout_click

이벤트를 쌓아두면 나중에 아래 비교가 가능하다.
- 상단 CTA vs 하단 CTA 중 뭐가 잘 눌리는가
- 가격 섹션 본 사람과 안 본 사람의 제출률 차이
- 모바일 유입과 PC 유입의 폼 제출률 차이

#### 4) 세그먼트 비교
사용자 그룹별 차이를 볼 수 있다.

예시:
- 모바일 vs 데스크톱
- 채널 A vs 채널 B
- 특정 캠페인 유입 vs 직접 유입

이게 중요한 이유:
랜딩은 전체 평균보다 세그먼트별 문제를 봐야 개선이 빠르다.

예를 들어 전체 전환율은 보통인데,
모바일 전환율만 매우 낮다면 디자인 또는 폼 UX 문제일 가능성이 높다.

#### 5) 가설 검증
랜딩페이지는 결국 가설 검증 도구다.

예를 들어 이런 가설을 세울 수 있다.
- 가격을 첫 화면에 보이면 오히려 폼 제출률이 높아질 것이다
- FAQ를 추가하면 결제 클릭률이 올라갈 것이다
- CTA 문구를 바꾸면 클릭률이 좋아질 것이다

PostHog가 있으면 이 변경 전후를 비교할 수 있다.
즉, 감으로 고치는 것이 아니라 데이터 기반으로 고칠 수 있다.

---

### 3-4. PostHog 없이 운영하면 생기는 문제
아래 문제가 자주 생긴다.

#### 문제 1. 방문자는 있는데 왜 안 되는지 모름
방문자 수만 보면 페이지가 먹히는지 아닌지 모른다.

#### 문제 2. 카피 문제인지, 폼 문제인지, 결제 문제인지 구분 못함
모든 문제가 "전환이 안 됨"으로 뭉개진다.

#### 문제 3. 개선 우선순위를 못 정함
버튼을 바꿔야 하는지,
FAQ를 넣어야 하는지,
폼을 줄여야 하는지 판단이 안 선다.

#### 문제 4. 채널이 달라도 같은 방식으로 해석함
실제로는 블로그 유입과 SNS 유입의 행동 패턴이 다를 수 있다.

---

### 3-5. PostHog를 처음 쓸 때 최소한으로 알아야 할 개념

#### 이벤트(Event)
사용자의 특정 행동 기록
예: 버튼 클릭, 폼 제출

#### 퍼널(Funnel)
여러 이벤트를 순서대로 놓고,
각 단계에서 얼마나 다음 단계로 넘어가는지 보는 것

#### 세션 리플레이(Session Replay)
실제 사용자가 페이지에서 어떻게 움직였는지 다시 보는 기능

#### 프로퍼티(Property)
이벤트나 사용자에 붙는 추가 정보
예: utm_source, device type, button 위치

#### 세그먼트(Segment)
특정 조건의 사용자 묶음
예: 모바일 사용자, 특정 캠페인 유입 사용자

---

### 3-6. 지금 랜딩에서 어떤 이벤트를 만들면 되나
초기에는 복잡하게 하지 말고 아래 정도면 충분하다.

#### 필수 이벤트
- page_view
- cta_click
- form_start
- form_submit
- thank_you_view
- checkout_click

#### 선택 이벤트
- scroll_50
- scroll_90
- faq_click
- email_click

#### 이벤트에 같이 남기면 좋은 속성
- CTA 위치(hero / middle / bottom)
- 페이지 URL
- UTM source
- UTM campaign
- device type

이렇게 해두면 단순 클릭 수보다 훨씬 많은 해석이 가능하다.

---

### 3-7. PostHog로 실제로 어떤 질문에 답할 수 있나
아래 질문들에 답할 수 있으면 랜딩 운영이 훨씬 쉬워진다.

- 첫 화면 메시지가 먹히는가?
- 어느 CTA가 가장 잘 눌리는가?
- 폼이 너무 길어서 이탈하는가?
- 모바일에서만 이상하게 이탈하는가?
- 특정 유입 채널이 다른 채널보다 제출률이 높은가?
- 가격을 본 사람이 더 잘 전환하는가?
- Thank you 페이지에서 결제 버튼을 거의 안 누르는가?

즉, PostHog는 단순 설치용 도구가 아니라,
랜딩을 반복 개선하기 위한 운영 도구다.

## 4. SEO / GEO는 어느 정도까지 준비하면 되나

초기에는 과하게 하지 않아도 된다.
다만 기본은 갖춰야 한다.

### SEO 최소 항목
- 페이지 제목
- 설명문
- OG 썸네일
- robots.txt
- sitemap.xml
- canonical

### GEO 관점의 핵심
AI나 검색 시스템이 페이지를 이해하기 쉽게 만들어야 한다.
즉 문장이 불명확하면 안 된다.

좋은 상태 예시:
- 누구를 위한 서비스인지 명확하다
- 얼마인지 명확하다
- 어떤 결과를 주는지 명확하다
- 신청 후 절차가 명확하다
- FAQ가 있다

## 5. 실제 배포 직전 최종 점검표

### A. 메시지 / 전환 구조
- [ ] 헤드라인이 누구를 위한 서비스인지 바로 보인다
- [ ] 가격이 명확히 공개되어 있다
- [ ] CTA가 명확하다
- [ ] 제공 내용이 구체적이다
- [ ] FAQ가 있다

### B. 폼 / 리드 수집
- [ ] 폼 항목이 최소화되어 있다
- [ ] 제출 데이터가 저장된다
- [ ] 자동 안내 메일이 간다
- [ ] 운영자에게 알림이 간다

### C. 결제 유도
- [ ] Thank you 페이지가 있다
- [ ] Thank you 페이지에 결제 버튼이 있다
- [ ] 메일에도 결제 링크가 있다
- [ ] 결제 링크가 정상 작동한다

### D. 추적 / 분석
- [ ] PostHog 설치 완료
- [ ] 핵심 이벤트 수집 확인
- [ ] 세션 리플레이 확인
- [ ] 퍼널 생성 완료
- [ ] UTM 저장 확인

### E. 검색 / 공유
- [ ] title 설정
- [ ] meta description 설정
- [ ] OG 썸네일 설정
- [ ] favicon 적용
- [ ] robots.txt 설정
- [ ] sitemap.xml 생성

### F. 안정성
- [ ] 모바일 테스트 완료
- [ ] 폼 제출 테스트 완료
- [ ] 메일 수신 테스트 완료
- [ ] 버튼 및 링크 전수 확인
- [ ] 개인정보 처리방침 링크 연결
- [ ] 에러 트래킹 가능 상태

## 6. 우선순위 정리

지금 단계에서 우선순위는 아래 순서가 맞다.

### 1순위
전환 동선이 실제로 작동하는가
- 폼 제출
- 자동 메일
- Thank you 페이지
- 결제 링크

### 2순위
어디서 이탈하는지 볼 수 있는가
- PostHog
- 핵심 이벤트
- 세션 리플레이
- 퍼널

### 3순위
유입 출처를 구분할 수 있는가
- UTM

### 4순위
검색/공유 기본기가 되어 있는가
- SEO / GEO

### 5순위
운영 중 오류를 잡을 수 있는가
- 에러 트래킹
- QA

## 7. 한 줄 결론

이 랜딩페이지의 배포 준비는 단순히 페이지를 완성하는 작업이 아니다.

핵심은 아래다.

- 전환 동선을 완성하고
- PostHog로 행동 데이터를 남기고
- 어디서 이탈하는지 확인하고
- 그 데이터를 바탕으로 반복 개선할 수 있게 만드는 것

특히 지금 단계에서 PostHog는 있으면 좋은 옵션이 아니라,
랜딩페이지를 실험 도구로 쓰기 위한 핵심 운영 도구에 가깝다.
